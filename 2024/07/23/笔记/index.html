Linux网络编程

一请求一线程

创建套接字之后，需要使用 `bind()` ​将套接字绑定到一个地址和端口上，然后使用 `listen()`​（对于服务器端）来等待客户端的连接请求，或者使用 `connect()`​（对于客户端）来与服务器建立连接。随后，可以使用 `accept()`​、`read()`​、`write()`​、`send()`​、`receive()` ​等函数来进行数据的传输。最后，当通信完成时，需要使用 `close()` ​来关闭套接字，释放资源。

socket ​函数

​fd 是 int 值，依次增加，系统内置三个 fd，所以 sockfd 从 3 开始

这个函数的参数有：

  • ​domain​：指定套接字的通信域。
    AF_INET 表示 IPv4 网络协议，AF_INET6 表示 IPv6 网络协议。除此之外，还有其他几种通信域，如 AF_UNIX 用于同一台机器上的进程间通信。

  • ​type​：指定套接字类型。
    SOCK_STREAM 表示字节流套接字，通常用于 TCP 协议；
    SOCK_DGRAM 表示数据报套接字，通常用于 UDP 协议；
    SOCK_RAW 允许程序直接访问网络层，一般用于特殊协议的开发。

  • ​protocol​：指定要使用的特定协议。通常，这个参数设为 0，以选择 domain ​和 type ​参数指定的默认协议。

    函数的返回值是一个文件描述符，用于在后续的套接字操作中引用这个新创建的套接字。如果 socket ​调用失败，它将返回-1，并设置全局变量 errno ​来表示错误原因。

sockaddr_in 结构

​​

  1. ​sin_family​：地址族。对于 sockaddr_in​，它通常被设置为 AF_INET​，表示该结构体用于 IPv4 网络地址。
  2. ​sin_port​：端口号。这是一个 16 位的数值，它存储了网络通信中使用的端口号。网络字节顺序（大端序）通常用于存储这个值，因此在使用之前需要用 htons()​ 函数将其从主机字节顺序转换为网络字节顺序。
  3. ​sin_addr​：IP 地址。这是一个 in_addr​ 结构体，用于存储 32 位的 IPv4 地址。通常使用 htonl(INADDR_ANY)​ 来指定服务器可以绑定到任何可用的本地地址，或者可以使用具体的 IP 地址来代替 INADDR_ANY​。
  4. ​sin_zero​：填充字段。这是一个 8 字节的数组，用于将 sockaddr_in​ 结构体填充到与 sockaddr​ 结构体相同的长度，以便可以强制转换这两种类型。通常，这个字段被设置为 0。​
  5. ​htonl()​ 和 htons() ​函数用于将主机字节顺序转换为网络字节顺序。在网络中，数据以标准字节顺序（大端序）发送，而主机可能使用大端序或小端序。htonl() ​函数转换 32 位，而 htons() ​函数转换 16 位。

bind、listen、accept

​​

使用 `bind()` ​将套接字绑定(设置)到一个地址和端口上，然后使用 `listen()`​（对于服务器端）来等待客户端的连接请求。

accept 函数

​`accept`​ 函数是用来接收一个传入的连接请求的。当一个服务器 socket 监听某个端口时，`accept`​ 函数会**阻塞**直到有一个客户端尝试连接到这个端口。当有连接请求时，`accept`​ 函数会创建一个新的 socket（通常称为连接 socket 或客户端 socket），用于与客户端进行通信，并返回这个新 socket 的文件描述符。

以下是 `accept`​ 函数的声明和参数说明：

参数解释：
  • ​sockfd​：这是服务器 socket 的文件描述符，它必须已经通过 socket​ 函数创建，并且已经通过 bind​ 绑定到一个地址和端口，以及通过 listen​ 函数进行了监听。
  • ​addr​：这是一个指向 sockaddr​ 结构的指针，用于存储发起连接的客户端的地址信息。这个参数通常是一个指向 sockaddr_in​ 结构的指针，然后通过类型转换使其符合 sockaddr​ 指针的类型。如果对客户端的地址不感兴趣，可以传递 ​NULL​​。
  • ​addrlen​：这是一个指向 socklen_t​ 类型的指针，用于指定 addr​ 结构的大小。在调用 accept​ 之前，它通常被设置为指向一个变量，该变量包含 addr​ 结构的大小。当 accept​ 函数返回时，这个变量会被更新为实际写入 addr​ 结构的字节数。

select ​函数在内核中执行以下步骤：

  1. 参数验证：首先，内核会验证传递给 select ​的参数，包括文件描述符集（fd_set​）和超时值。

  2. 复制文件描述符集：内核会将用户空间中的 fd_set ​结构复制到内核空间。这是因为 select ​需要在内核态中操作这些集合，并且不能直接访问用户空间的数据。

  3. 设置监听：内核会对 fd_set ​中的每个文件描述符设置监听。对于 &rset​（读集），内核会检查这些文件描述符是否有数据可读；对于 &wset​（写集），会检查是否可以写入数据；对于 &eset​（异常集），会检查是否有异常条件发生。

  4. 阻塞和等待：如果没有任何文件描述符立即就绪，select ​将使调用进程进入睡眠状态，直到以下任一情况发生：

    • 至少一个文件描述符变为就绪状态。
    • 调用被信号处理程序中断。
    • 超时发生。

  5. 就绪处理：当至少一个文件描述符变为就绪状态时，内核会更新相应的 fd_set​，标记哪些文件描述符已经就绪。

  6. 返回用户空间：select ​调用结束后，内核会将更新后的 ​fd_set​ 结构复制回用户空间，并将就绪文件描述符的数量作为函数返回值。

select 内核具体更新 fd_set 的流程：

  • 检查文件描述符状态： vfs_poll​ 函数返回 mask​，表示文件描述符的状态。

    ​vfs_poll​ 函数的大致流程如下：

    1. 获取文件的 ​poll​​方法： 每个文件对象都有一个 file_operations​ 结构，这个结构包含了文件操作的各种函数指针，包括 poll​。
    2. 调用 ​poll​​方法： vfs_poll​ 函数调用文件对象的 poll​ 方法，将 poll​ 表传递给它。
    3. 返回文件状态： 文件的 poll​ 方法返回一个位掩码，这个掩码表示文件的状态。

  • 设置结果位图： 根据 mask​ 更新局部变量 res_in​, res_out​, res_ex​。如果文件描述符不可读（或不可写、不可发生异常），对应位不设置为 1，这相当于保持为 0。

  • 写回结果位图： 将局部变量 res_in​, res_out​, res_ex​ 中的内容写回到结果位图 rinp​, routp​, rexp​。

Linux select 函数的缺点也是显而易见的：

  • 每次调用 select 函数，都需要把 fd 集合从用户态拷贝到内核态，这个开销在 fd 较多时会很大，同时每次调用 select 函数都需要在内核遍历传递进来的所有 fd，这个开销在 fd 较多时也很大；
  • 单个进程能够监视的文件描述符的数量存在最大限制，在 Linux 上一般为 1024，可以通过修改宏定义然后重新编译内核的方式提升这一限制，这样非常麻烦而且效率低下；
  • select 函数在每次调用之前都要对传入参数进行重新设定，这样做也比较麻烦。

底层原理：epoll 源码分析，还搞不懂 epoll 的看过来.md

​​

epoll 底层实现中有两个关键的数据结构，一个是 eventpoll 另一个是 epitem，其中 eventpoll 中有两个成员变量分别是 rbr 和 rdlist,前者指向一颗红黑树的根，后者指向双向链表的头。而 epitem 则是红黑树节点和双向链表节点的综合体，也就是说 epitem 即可作为树的节点，又可以作为链表的节点，并且 epitem 中包含着用户注册的事件。

当用户调用 epoll_create()时，会创建 eventpoll 对象（包含一个红黑树和一个双链表）；

​​

    1. 创建 eventpoll 对象
    2. 让 eventpoll 中的 rbr 指向空
    3. 让 eventpoll 中的 rdlist 指向空
    4. 在并发环境下进行互斥
    5. 保存 eventpoll 对象
    6. 返回 eventpoll 对象的句柄(id)

而用户调用 epoll_ctl(op)时，

该函数的逻辑其实很简单，无非就是将用户传入的参数封装为一个 epitem 对象，然后根据传入的 op 是
①EPOLL_CTL_ADD、②EPOLL_CTL_MOD 还是 ③EPOLL_CTL_DEL，来决定是
① 将 epitem 对象插入红黑树中，② 更新红黑树中的 epitem 对象，还是 ③ 移除红黑树中的 epitem 对象。

epoll_event_callback()

操作系统会默默地在通过 epoll_event_callback()来管理 eventpoll 对象。当有事件被触发时，操作系统则会调用 epoll_event_callback 函数，将含有该事件的 epitem 添加到双向链表中。

当服务器在以下 5 种情况会调用 epoll_event_callback：

  1. 客户端 connect()连入，服务器处于 SYN_RCVD 状态时
  2. 三路握手完成，服务器处于 ESTABLISHED 状态时
  3. 客户端 close()断开连接，服务器处于 FIN_WAIT_1 和 FIN_WAIT_2 状态时
  4. 客户端 send/write()数据，服务器可读时
  5. 服务器可以发送数据时

当用户需要管理连接时，只需通过 epoll_wait()从 eventpoll 对象中的双链表下”摘取”epitem 并取出其包含的事件即可。

黏包问题

  • 黏包是 TCP 通信中常见的现象，主要由于 TCP 的流模式及发送和接收的不匹配导致。

  • 常见的解决方法包括使用定长消息、分隔符或消息头中包含长度字段。

  • 水平触发方式在处理黏包问题时更为可靠，因为它会持续通知应用程序处理剩余数据。

    边缘触发只会通知应用程序一次，即数据到达时。应用程序需要在这次通知中尽可能多地读取数据。如果数据没有完全读取完毕（因为数据量很大或者读取过程中被打断），应用程序不会再收到通知，除非有新的数据到达。
